# RPA safety rules — rebuild (canonical)

**Product alignment:** Liquor Kings is a **control and reconciliation** system; browser RPA here is one execution mechanism alongside **MILO** (current production lane). Partner portals (MLCC, SIPS+, successors) are expected to change; safety rules and verify gates remain binding until an **explicitly approved** phase or doc change updates them. See [strategic-architecture.md](./strategic-architecture.md).

**Non-negotiable:** Until an **explicitly approved later phase**, the MLCC browser dry-run / rebuild path must **not** perform:

- Real order **submission**
- **Checkout** or final confirmation flows
- Cart **validate** (MLCC “validate order” style actions) — **forbidden** except **Phase 2q** when **both** `MLCC_ADD_BY_CODE_PHASE_2Q` and `MLCC_ADD_BY_CODE_PHASE_2Q_APPROVED` are true, **2n**/**2l** gates match **2n** requirements, probe is on, `MLCC_ADD_BY_CODE_PHASE_2Q_VALIDATE_SELECTORS` is non-empty, and (when **2o** is off) `MLCC_ADD_BY_CODE_PHASE_2Q_OPERATOR_ACCEPTS_MISSING_2O=true` — **at most one** validate click; **no** checkout, submit, finalize, or second validate
- **Add-to-cart** or other **cart/order mutation** paths — **forbidden** except **Phase 2n**’s **single** tenant-listed add-line / apply-line click when fully env-gated (still **no** add-to-cart, validate, checkout, or submit in worker code paths outside documented phases)
- **Quantity** field entry except **Phase 2j** when **both** `MLCC_ADD_BY_CODE_PHASE_2J` and `MLCC_ADD_BY_CODE_PHASE_2J_APPROVED` are true and tenant quantity selector + valid test quantity are configured (quantity field **only**; no code interaction in that phase)
- **Combined** code+quantity interaction — **forbidden** except **Phase 2l** when **both** `MLCC_ADD_BY_CODE_PHASE_2L` and `MLCC_ADD_BY_CODE_PHASE_2L_APPROVED` are true and valid test code, test quantity, `MLCC_ADD_BY_CODE_PHASE_2L_FIELD_ORDER`, and both tenant selectors are configured (**no** add line inside **2l**, **no** validate/checkout/submit).
- **Add-line / apply-line** (or equivalent pre-cart commit) **clicks** — **forbidden** except **Phase 2n** when **both** `MLCC_ADD_BY_CODE_PHASE_2N` and `MLCC_ADD_BY_CODE_PHASE_2N_APPROVED` are true, **2l** is enabled and approved in the same config, probe is on, and `MLCC_ADD_BY_CODE_PHASE_2N_ADD_APPLY_SELECTORS` is a non-empty tenant list — **at most one** such click; **no** validate, checkout, submit, or second apply in **2n**. **Phase 2m** is the **canonical manifest** (probe import only; worker must not import **2m**).
- **Post-add/apply read-only observation** — **forbidden** except **Phase 2o** when **both** `MLCC_ADD_BY_CODE_PHASE_2O` and `MLCC_ADD_BY_CODE_PHASE_2O_APPROVED` are true, **2n** gates are satisfied, and probe is on: **zero** UI clicks (read-only scrapes + optional settle wait only); **no** validate, checkout, submit, or second add/apply; evidence must **not** claim server cart or inventory truth from DOM alone.
- **Post-validate read-only observation** — **forbidden** except **Phase 2r** when **both** `MLCC_ADD_BY_CODE_PHASE_2R` and `MLCC_ADD_BY_CODE_PHASE_2R_APPROVED` are true, **2q** is enabled and approved and completed with `validate_click_performed` in the same run (and **2q** Layer 2 prerequisite satisfied), and probe is on: **zero** UI clicks; read-only scrapes + optional settle; may record **inferred** visible checkout/submit-adjacent control samples from text/href heuristics **without clicking**; evidence must **not** claim backend order truth, safe checkout, or submit readiness from DOM alone.

Violations are a **product incident**, not a documentation gap.

## Three-layer blocking (current implementation)

1. **Code paths** — Worker does not implement submit/checkout/finalize or add-to-cart. **Phase 2n** is the only path that may click add-line / apply-line (env-gated, at most once). **Phase 2q** is the only path that may click **validate** (env-gated, at most one tenant-listed control; **no** checkout/submit/finalize). **Phase 2o** performs **no** clicks — read-only observation after a successful **2n** click only. **Phase 2r** performs **no** clicks — read-only observation after a successful **2q** validate click only (inferred checkout-like visibility scan is **not** a click). Success notes distinguish combined rehearsal clears, **2n**, **2o**, **2q**, and **2r**. `assertMlccSubmissionAllowed` exists for **future** submit code and **throws** unless `MLCC_SUBMISSION_ARMED=true` (currently unused in the happy path).
2. **Network** — `installMlccSafetyNetworkGuards` + `shouldBlockHttpRequest` in `mlcc-browser-add-by-code-probe.js` aborts matching mutation-like URLs (methods/patterns as coded). **Validate-shaped mutation URLs are not in the default block list** so **2q** can observe real validate requests where applicable; checkout/submit/cart-add patterns stay blocked.
3. **UI** — `MLCC_PROBE_UNSAFE_UI_TEXT` / `isProbeUiTextUnsafe` block checkout/add-to-cart and similar labels on **generic** probe clicks; **validate** labels are **not** globally blocked so **2q** can use `evaluatePhase2qValidateCandidateEligibility` + tenant selectors. **Add-line / apply-line** labels are **not** globally blocked so **2n** uses `evaluatePhase2nAddApplyCandidateEligibility`. Phase 2d/2e classify controls (2e may scope to a root selector) but **do not click**. Phase **2f** may perform **at most one** click on a tenant-listed candidate only after the same Layer 3 + mutation-boundary eligibility checks. Phase **2g** adds field policy + extended DOM risk readout; optional **focus/blur** or **synthetic sentinel** fill/clear only when env-gated — **no** real product codes or quantities. Phase **2h** may **fill** a single tenant **code** field only when **both** `MLCC_ADD_BY_CODE_PHASE_2H` and `MLCC_ADD_BY_CODE_PHASE_2H_APPROVED` are true and a valid test code is configured — **no** Enter, **no** quantity field, **no** submit/add/validate/checkout clicks; **hard-fail** if Layer 2 abort count increases during the type step. Phase **2j** may **fill** a single tenant **quantity** field only when **both** `MLCC_ADD_BY_CODE_PHASE_2J` and `MLCC_ADD_BY_CODE_PHASE_2J_APPROVED` are true and a valid test quantity is configured — **no** code field interaction, **no** Enter, **no** submit/add/validate/checkout clicks; optional blur only if `MLCC_ADD_BY_CODE_PHASE_2J_ALLOW_BLUR=true`; **hard-fail** if Layer 2 abort count increases during the type step (same clear policy as **2h**). Phase **2l** may perform the **combined** code+quantity **fill** sequence (tenant `MLCC_ADD_BY_CODE_PHASE_2L_FIELD_ORDER` only) when **both** `MLCC_ADD_BY_CODE_PHASE_2L` and `MLCC_ADD_BY_CODE_PHASE_2L_APPROVED` are true — **no** Enter, **no** add/apply-line/validate/checkout/submit clicks; optional blur on the last filled field only if `MLCC_ADD_BY_CODE_PHASE_2L_ALLOW_BLUR=true`; **hard-fail** if Layer 2 abort count increases during **any** fill step (fields not cleared). Phase **2n** may perform **at most one** click on a tenant-listed add/apply control only after successful **2l** in the same run and passing Layer 2/3 + eligibility checks — **no** validate/checkout/submit or second click; **hard-fail** if Layer 2 abort count increases during the click window. Phase **2o** performs **no** clicks — read-only DOM/status capture after **2n**; **hard-fail** if Layer 2 abort count increases during the observation window. Phase **2q** may perform **at most one** click on a tenant-listed validate control after **2n** (and **2o** when enabled, or operator waiver when **2o** off); optional read-only post-validate scrape only — **no** checkout/submit/finalize/second validate; **hard-fail** if Layer 2 abort count increases during the declared **2q** window.

`MLCC_BROWSER_DRY_RUN_SAFE_MODE` is exported as `true` from `mlcc-browser-worker.js`.

## Hard-fail behavior

- Worker failures finalize runs with `failure_type` / `failure_details` / `evidence` as implemented.
- Phase 2c/2d/2e/2f/2g/2h/2j/2l/2n/2o/2q/2r errors are wrapped with identifiable messages (`MLCC add-by-code phase 2c failed`, `… phase 2h failed`, `… phase 2j failed`, `… phase 2l failed`, `… phase 2n failed`, `… phase 2o failed`, `… phase 2q failed`, `… phase 2r failed`, etc.) for operator diagnostics.

## Truthfulness

- Evidence and diagnostics must not claim precision beyond stored fields (`failure_details`, attempts, evidence attributes). MLCC signals may be explicit or inferred per `mlcc-operator-context.service.js`.

## Future quantity and post-quantity ladder (Phase 2i / 2j)

Phase **2i** is **planning-only** as a standalone phase (no `PHASE_2I` env): it is the versioned gate manifest and ladder in [`services/api/src/workers/mlcc-phase-2i-policy.js`](../../../services/api/src/workers/mlcc-phase-2i-policy.js). **Phase 2j** is the first **runtime** quantity rehearsal; it **imports** that manifest for evidence and stays **quantity-field-only** (no code field, no combined code+qty step, no add-to-cart/validate/checkout/submit).

- **Canonical criteria** for quantity and later steps: `buildPhase2iQuantityFutureGateManifest`, `buildPhase2iBroaderInteractionLadder`. Summaries appear in [rpa-rebuild-phases.md](./rpa-rebuild-phases.md).
- **Ladder:** after **2j**, remaining steps (add/apply, validate, checkout/submit, etc.) stay **`out_of_scope_until_separate_approval`** until explicitly documented and verified.
- **Non-negotiables:** outside **2j**’s env-gated quantity fill/clear, quantity entry and cart/order mutation paths remain forbidden as documented in each phase row.

## Combined interaction planning (Phase 2k) and runtime rehearsal (Phase 2l)

Phase **2k** is the versioned checklist in [`services/api/src/workers/mlcc-phase-2k-policy.js`](../../../services/api/src/workers/mlcc-phase-2k-policy.js) (`buildPhase2kCombinedInteractionFutureGateManifest`, `buildPhase2kPostCombinedInteractionLadder`). It **does not** run standalone in the browser; the **probe** imports it to echo the manifest during **2l**. The **worker** must **not** import **2k** (verify enforces).

- **Criteria (2k / 2l):** both tenant selectors, **tenant-documented field order** via `MLCC_ADD_BY_CODE_PHASE_2L_FIELD_ORDER`, extended mutation-risk on **both** fields before fills and **after the first fill**, Layer 2 deltas zero per fill and clear step, Layer 3 no add-line/apply/cart/validate/checkout/submit clicks, hard-fail parity with **2h**/**2j** on fill aborts, truthful disclaimers (no general combined safety, no add-line readiness, no server cart proof).
- **Ladder after 2l:** **2n** implements the single add/apply-line click when env-gated; **2o** implements read-only post-click observation when env-gated; **2q** implements the single validate click when env-gated; checkout/submit remain **`out_of_scope_until_separate_approval`**; **Phase 2m** is the canonical add/apply manifest (probe import only; see below).

## Add/apply-line manifest (Phase 2m), runtime click (Phase 2n), and read-only observation (Phase 2o)

[`services/api/src/workers/mlcc-phase-2m-policy.js`](../../../services/api/src/workers/mlcc-phase-2m-policy.js) exports `buildPhase2mAddApplyLineFutureGateManifest` and `buildPhase2mPostAddApplyLadder`. The **probe** imports **2m** to echo the manifest and ladder during **2n** and **2o**; the **worker** must **not** import **2m** (verify enforces).

- **Criteria (2m / 2n / 2o):** successful **2l** in the same run before **2n**; tenant-listed add/apply control selector(s); **at most one** add/apply click in **2n**; **2o** only after **2n** `add_apply_click_performed`; **2o** uses **zero** clicks and truthful disclaimers (DOM ≠ server cart; regex hints are inference only).
- **Ladder:** `add_apply_line_rehearsal` is **implemented** as **2n** when env-gated; `post_add_apply_observation` is **implemented** as **2o** when env-gated; **`validate_order`** is **implemented** as **2q** when env-gated; **checkout/submit** on the **2m** ladder remain **`out_of_scope_until_separate_approval`**. **Validate** manifest and post-validate ladder detail: **2p** (below).

## Validate manifest (Phase 2p), bounded validate (Phase 2q), and read-only post-validate observation (Phase 2r)

[`services/api/src/workers/mlcc-phase-2p-policy.js`](../../../services/api/src/workers/mlcc-phase-2p-policy.js) exports `buildPhase2pValidateFutureGateManifest` and `buildPhase2pPostValidateLadder`. **Phase 2p** has **no** `MLCC_ADD_BY_CODE_PHASE_2P` env flag. The **probe** imports **2p** to echo the manifest and ladder during **2q** and **2r**; the **worker** must **not** import **2p** (verify enforces).

- **Criteria (2p / 2q):** same-run **2n** with `add_apply_click_performed`; when **2o** is enabled, same-run **2o** with `observation_performed` and Layer 2 prerequisites; when **2o** is off, explicit `MLCC_ADD_BY_CODE_PHASE_2Q_OPERATOR_ACCEPTS_MISSING_2O=true`; non-empty tenant **`MLCC_ADD_BY_CODE_PHASE_2Q_VALIDATE_SELECTORS`**; **at most one** validate click; **hard-fail** if Layer 2 blocked-request count increases over the **2q** window; optional read-only post-validate scrape only; truthful disclaimers (browser/client evidence ≠ backend order truth; no general validate safety; no checkout readiness).
- **Criteria (2p / 2r):** same-run **2q** with `validate_click_performed` and **2q** Layer 2 prerequisite not failed; **zero** clicks; pre/post read-only scrapes with optional settle; inferred checkout-adjacent visible control list is **observation only** — **no** interaction; **hard-fail** if Layer 2 blocked-request count increases over the **2r** window; disclaimers that visible “Checkout”-like labels do **not** prove safe checkout or backend order state.
- **Post-validate ladder:** `validate_order_bounded_interaction` is **`implemented_as_phase_2q_when_env_gated`**; `post_validate_observation` is **`implemented_as_phase_2r_when_env_gated`**; `checkout_flow` and `submit_finalize_order` remain **`out_of_scope_until_separate_approval`** until future phases.
- **Non-negotiable:** **No** checkout, submit, or order finalization in **2q** or **2r**; **do not** assume MLCC validate or post-validate DOM implies generally safe checkout.

## Checkout planning (Phase 2s)

Phase **2s** is **planning-only**: [`services/api/src/workers/mlcc-phase-2s-policy.js`](../../../services/api/src/workers/mlcc-phase-2s-policy.js) exports `buildPhase2sCheckoutFutureGateManifest` and `buildPhase2sPostCheckoutLadder`. It **does not** run in the browser; the **worker** and **probe** **must not** import **2s** until a separately approved **checkout execution** phase (e.g. **2t**) updates `verify:lk:rpa-safety`.

- **Criteria (manifest):** evidence expectations after **2q** / **2r** (or documented equivalent when **2r** is off); future non-heuristic tenant checkout selector list; Layer 2 notes (checkout-shaped URLs may remain blocked until an execution phase documents policy); Layer 3 / mutation-boundary family expectations for a future bounded click; hard-fail stops; observable proof criteria; **mandatory_disclaimers** (browser/client signals ≠ backend checkout or payment truth).
- **Post-checkout ladder:** `checkout_bounded_interaction` → `post_checkout_observation` → `submit_finalize_order` — each **`out_of_scope_until_separate_approval`** until implemented.
- **Non-negotiable:** **No** runtime checkout, submit, or finalize in **2s** itself; **do not** treat **2r** inferred checkout-like controls as approval to implement checkout without a new gated phase.

## Drift enforcement

- Run `npm run verify:lk:rpa-safety` from repo root (includes Phase **2i**, **2k**, **2m**, **2p**, and **2s** policy files, phases-doc markers, Phase **2l**/**2n**/**2o**/**2q**/**2r** probe/worker markers, probe import of **2k**, **2m**, and **2p**, guards that the **worker** does not import **2k**, **2m**, **2p**, or **2s**, and guards that the **probe** does not import **2s** until a checkout execution phase).
- **Operational repeatability (no new runtime risk):** [mlcc-dry-run-repeatability.md](./mlcc-dry-run-repeatability.md) documents env/checklists; `npm run doctor:lk:mlcc-dry-run` validates config for safe phases **without** launching a browser or checkout.
- See [DEVELOPER_ANTI_DRIFT.md](../DEVELOPER_ANTI_DRIFT.md).
